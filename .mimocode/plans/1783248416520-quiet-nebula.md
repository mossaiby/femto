# Plan: Build the femto Compiler

## Overview

Build a complete compiler for the femto language (spec in `SPECS.md`) targeting x86-64 Linux with NASM-compatible Intel syntax assembly output. Implemented in C++26, using only `std` dependencies. CMake build system, GoogleTest for unit tests, `.femto` integration tests.

## Implementation Phases

### Phase 1: Project Skeleton + Lexer (Week 1)

**Files:** `CMakeLists.txt`, `src/common/*`, `src/lexer/*`

1. Set up CMake project with C++26, GoogleTest
2. Implement `SourceLocation` (file, line, column, span)
3. Implement `Diagnostic` engine (errors/warnings with source context)
4. Implement `StringPool` for string interning
5. Implement `Lexer`:
   - All token types (keywords, operators, literals, delimiters)
   - `//` and `/* */` (nested) comments
   - Integer literals: decimal, hex `0xFF`, binary `0b1010`, octal `0o755`, digit separator `_`
   - Float literals: `3.14`, `1.0e-9`
   - Char literals: `'a'`, `'\n'`, `'\u{1F600}'`
   - String literals: `"hello"`, raw strings `` `...` ``
   - Bool literals: `true`, `false`
   - All 39 keywords
   - `@`-builtins: `@target`, `@arch`, `@endian`, `@file`, `@line`, `@sizeof`, `@alignof`, `@typeof`, `@bitcast`
   - `#export`, `#if`, `#else` as preprocessing tokens
6. Unit tests for lexer

### Phase 2: Parser + AST (Week 2-3)

**Files:** `src/parser/*`

1. Define complete AST node hierarchy (all node types from spec):
   - Declarations: variables, constants (`::`), functions, structs, enums, unions, namespaces
   - Expressions: literals, identifiers, binary/unary ops, function calls, casts, generic instantiations, `??` operator, `success`/`failure`
   - Statements: assignments, `if`/`then`/`else`, `while`, `do`/`while`, `switch`, `match`, `foreach`, `break`/`continue`, `return`
   - Types: primitive types, pointers `T*`, fixed arrays `T[N]`, slices `T[]`, function types `!T`
   - Preprocessor: `#if`, `#else`, `#export`
2. Recursive descent parser:
   - All operator precedences (C-like)
   - `then` keyword requirement after `if` conditions
   - Block requirements for control flow
   - Generics: `<T>`, `<K, V>` syntax on functions and structs
   - Default arguments (trailing)
   - Function overloading declarations
   - `extern "C"` blocks
   - `match` expression syntax with `#` subject binding
   - `foreach` with optional index
   - `break(n)` / `continue(n)` with optional integer
3. Unit tests for parser

### Phase 3: Semantic Analysis (Week 3-4)

**Files:** `src/sema/*`

1. Symbol table with nested scopes (namespace, function, block)
2. Type system:
   - All primitive types with sizes
   - Pointer types, array types, slice types
   - Function types (with error variants `!T`)
   - Struct/enum/union types
   - Generic type parameters
3. Type checking:
   - No implicit widening/narrowing/signedness changes
   - Call-style casts with value-preservation check
   - `int8!(x)` for potentially-lossy casts
   - `@bitcast(T, x)` for reinterpreting
4. Overload resolution:
   - Arity and type matching
   - Default argument filling
   - Ambiguity detection
   - Generic vs non-generic preference
5. Definite assignment analysis
6. `failure` result value-field inaccessibility check
7. Module system:
   - `import` resolution
   - `#export` visibility
   - `extern "C"` validation (no generics/overloading)
8. Compile-time evaluation:
   - `::` constants
   - `#if` / `#else` condition evaluation
   - `@`-builtins resolution
9. Unit tests for sema

### Phase 4: HIR + HIR Optimizer (Week 4-5)

**Files:** `src/hir/*`

1. HIR representation (lowered from AST):
   - SSA-like form for expressions
   - Explicit types on all nodes
   - Monomorphized generics (concrete types resolved)
   - Simplified control flow
2. AST -> HIR lowering:
   - Generic instantiation
   - Default argument expansion
   - Overload resolution finalization
   - Implicit cast insertion (where allowed)
3. HIR Optimizer:
   - Constant folding (compile-time `::` values, `@`-builtins)
   - Dead code elimination
   - `#if` branch elimination (discarded branches still parsed but not in HIR)
4. Unit tests

### Phase 5: LIR + LIR Optimizer (Week 5-6)

**Files:** `src/lir/*`

1. LIR representation (target-agnostic):
   - Three-address code or similar
   - Virtual registers
   - Explicit memory operations
   - Function prologue/epilogue placeholders
2. HIR -> LIR lowering:
   - Break down complex expressions
   - Explicit stack allocation for locals
   - Slice operations to pointer+length
   - String literals to data section references
3. LIR Optimizer:
   - Peephole optimizations
   - Redundant load/store elimination
   - Simple register allocation hints
4. Unit tests

### Phase 6: x86-64 Code Generation (Week 6-8)

**Files:** `src/codegen/x86_64/*`, `src/codegen/linker.*`

1. System V AMD64 ABI calling convention
2. Register allocator (graph coloring or linear scan)
3. Instruction selection:
   - Integer arithmetic (all sizes 8-512 bits via extended instructions)
   - Floating point (SSE/AVX)
   - Comparisons and conditional moves
   - Memory operations (leas, loads, stores)
   - Stack frame management
   - Function calls (register passing, stack spills)
   - Variadic-like handling for `array<T>` operations
4. String literal emission (data section)
5. Struct layout (C-compatible)
6. Enum backing type handling
7. Union overlay
8. Pointer operations (address-of, dereference)
9. Slice operations (bounds checking)
10. Fixed-size array operations
11. `!T` result type codegen (struct with code+value)
12. `??` operator codegen (branch on code)
13. Error propagation (`??` with early return)
14. NASM assembly output (Intel syntax)
15. Linker invocation: `nasm -f elf64` + `ld` with libc
16. Unit tests + integration tests

### Phase 7: Standard Library + Interop (Week 8-9)

**Files:** `src/std/*`, `extern` support

1. `std::io::print` (format string with `{}` placeholders)
2. `std::io::read` (file reading)
3. `std::mem::alloc<T>`, `std::mem::free`
4. `std::string8::*` functions (concat, repeat, slice, from_chars, from_int, from_float)
5. `std::collection::array<T>` (push, pop, length, capacity)
6. `extern "C"` function declarations (call libc functions)
7. Integration tests

### Phase 8: Polish + Testing (Week 9-10)

1. Comprehensive integration test suite:
   - Basic types and literals
   - All control flow constructs
   - Error handling (!T, ??, success/failure)
   - Generics (functions and structs)
   - Modules and imports
   - Extern interop
   - Compile-time features (#if, @-builtins)
2. Self-hosted test: write femto test programs in femto
3. Edge cases: nested comments, deeply nested generics, overload ambiguity
4. Error message quality improvements

## Critical Design Decisions

1. **AST representation**: Use `std::variant` with `std::unique_ptr` for child nodes. Each node type is a struct in a variant. This is type-safe and avoids vtable overhead.

2. **String handling**: All strings interned via `StringPool`. Token strings are pool indices.

3. **Type representation**: Use a `Type` variant in sema that can represent all types including generic parameters, pointers, arrays, function types.

4. **Error recovery**: Parser and sema should attempt error recovery (panic-mode) to report multiple errors per compilation.

5. **Monomorphization**: Generics are instantiated during HIR lowering after type checking. Each instantiation produces concrete HIR.

6. **Output format**: NASM-compatible x86-64 Intel syntax `.asm` files, assembled with `nasm -f elf64`, linked with `ld` (or `gcc` for libc).

## Verification

1. **Unit tests**: `ctest` runs all GoogleTest unit tests for each phase
2. **Integration tests**: Compile `.femto` files, run resulting binaries, compare stdout to expected output
3. **Smoke test**: Write a small femto program that exercises all features, compile and run it
4. **Self-hosted**: Eventually compile a femto program with the femto compiler

## Dependencies to Install

- NASM: `sudo apt install nasm` (required for x86-64 assembly)
- GoogleTest: fetched automatically by CMake (`FetchContent`)

## Estimated Timeline

~10 weeks for a full implementation. Each phase builds on the previous. Phases 1-3 (frontend) are the foundation; phases 4-6 (backend) are the most complex. Phase 7 (stdlib) and 8 (polish) wrap up.
